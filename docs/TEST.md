# 테스트 가이드

IntervAI의 통합 테스트 작성 시 재사용할 수 있는 기본 구조를 정리한다.
여기서는 특정 테스트 코드 전체를 복붙하지 않고, 다음 테스트를 같은 방식으로 설계할 수 있을 정도의 기준만 남긴다.

---

## 통합 테스트 기본 원칙

- 여러 API를 순서대로 호출하며 상태 전이를 검증하는 경우 `MockMvc` 단위 테스트보다 포트 기반 통합 테스트를 우선한다.
- 통합 테스트는 실제 HTTP 요청, 보안 필터, DB 상태 변경까지 함께 검증한다.
- 외부 연동(LLM 등)은 테스트 전용 구현으로 고정해서 테스트를 결정적으로 만든다.

---

## 권장 구조

### 1. 공통 베이스

- 공통 베이스 클래스는 랜덤 포트로 서버를 띄우고, HTTP 호출과 JSON 직렬화/역직렬화만 담당한다.
- Redis 같은 외부 상태 저장소를 쓰는 경우 테스트 종료 후 정리 메서드를 제공한다.
- 도메인별 step 객체를 생성하는 팩토리 메서드만 둔다.

예시 역할:
- `newHttpClient()`
- `sendJson(...)`
- `readBody(...)`
- `userSteps(client)`, `questionSteps(client)` 같은 step factory(캐싱 적용)

### 2. 도메인별 API Step 클래스

- 시나리오 단계는 하나의 큰 helper로 몰지 말고 도메인 패키지별로 나눈다.
- 각 step 클래스는 공통 `ApiSteps` 추상 클래스를 상속받아 `api`, `client`를 공유한다.
- 각 메서드는 “요청 1회 + 상태 코드 검증 + 응답 DTO 반환”까지만 담당한다.

패키지 예시:
- `user/integration` — 회원가입, 로그인
- `interview/integration` — 면접 생성
- `session/integration` — 세션 생성, 세션 종료
- `question/integration` — 질문 생성, 현재 질문 조회
- `answer/integration` — 답변 제출

메서드 네이밍 원칙:
- `signUp`
- `login`
- `createInterview`

번호는 실제 사용자 플로우 순서를 드러내기 위한 용도이며, 시퀀스 테스트에서 읽기 쉽게 유지한다.

### 3. 시퀀스 오케스트레이터 테스트

- 실제 시나리오 테스트 클래스는 각 step 객체를 조합해 전체 흐름을 실행한다.
- 테스트 본문에서는 “무엇을 검증하는 흐름인지”가 바로 보이도록 단계 순서가 드러나야 한다.
- 반복이 필요한 구간은 루프를 사용하되, 무한 반복 방지를 위한 최대 반복 횟수를 둔다.

예시 흐름:
1. 회원가입
2. 로그인
3. 면접 생성
4. 세션 생성
5. 질문 생성
6. 현재 질문 조회
7. 답변 제출
8. 종료 조건이 될 때까지 6~7 반복
9. 세션 종료

---

## 검증 범위

시퀀스 통합 테스트는 최소한 아래 두 층위를 함께 검증한다.

### 1. API 응답 검증

- 각 단계의 HTTP 상태 코드
- 필수 응답 필드 존재 여부
- 종료 조건(`hasNext`, `sessionStatus` 등) 변화

### 2. 저장 상태 검증

- 세션 상태가 기대한 값으로 바뀌었는지
- 질문/답변/피드백 개수가 기대와 맞는지
- 마지막 완료 시각 같은 후처리 필드가 기록됐는지

응답만 확인하고 끝내지 말고, 최종 상태는 repository 조회로 한 번 더 검증한다.

---

## 테스트 전용 구현체 사용 기준

- 질문 생성, 답변 피드백 생성처럼 외부 시스템에 의존하는 부분은 `@TestConfiguration` + `@Primary`로 덮어쓴다.
- 테스트 전용 구현체는 “이 시나리오가 어떤 상태 전이를 만들어야 하는지”를 명확히 드러내야 한다.
- 예를 들어 마지막 본 질문에서는 꼬리 질문을 만들지 않도록 고정하면 종료 시나리오를 안정적으로 검증할 수 있다.

권장 원칙:
- 랜덤하지 않을 것
- 네트워크에 의존하지 않을 것
- 종료 조건을 테스트가 제어할 수 있을 것

---

## 데이터 정리 원칙

- 테스트 종료 후 저장된 엔티티는 명시적으로 정리한다.
- `@Transactional` 롤백만 믿지 말고, HTTP 기반 통합 테스트에서는 repository `deleteAllInBatch()`와 Redis flush를 함께 고려한다.
- 정리 순서는 FK 제약을 고려해 하위 엔티티부터 상위 엔티티 순으로 지운다.

---

## 언제 이 구조를 쓰는가

아래 조건이면 이 구조를 우선 사용한다.

- API가 2개 이상 연속 호출되어야 의미가 생김
- 한 요청의 결과가 다음 요청 입력으로 사용됨
- 단순 컨트롤러 검증이 아니라 상태 전이가 핵심임
- 인증, 세션, 진행 상태, 완료 처리까지 함께 확인해야 함

반대로 단일 엔드포인트의 입력 검증, 에러 포맷, 상태 코드만 확인하면 되는 경우는 기존 `WebMvcTest`/`MockMvc` 테스트가 더 적합하다.
